# Aspira Pay 跨境支付系统完整架构设计文档

> 版本：v2.0  
> 说明：本版本加入个人账户注册、登录、KYC、银行卡申请、银行卡绑定、管理员/普通用户权限区分、用户信息管理等模块。  
> 备注：本版本暂不加入人脸识别能力。

---

## 1. 系统定位

Aspira Pay 是一个参考 Wise 类产品思路设计的跨境支付与多币种账户系统，目标是为个人用户、企业用户和平台管理员提供安全、可审计、可扩展的跨境支付、换汇、清算和账户管理能力。

系统核心能力包括：

- 个人账户注册、登录、身份认证
- KYC/AML 合规审核
- 多币种钱包账户
- 银行卡绑定、虚拟卡/实体卡申请与管理
- 跨境转账与本地支付
- 实时汇率换算
- 手续费计算
- 资金冻结、扣款、入账、退款
- 管理员后台
- 风控、审计、对账、清算
- 高并发交易处理
- 可扩展 API 接入

## 3. 用户角色设计

系统需要明确区分不同角色，保证普通用户、管理员、系统服务账号之间权限隔离。

### 3.1 普通用户

普通用户主要面向个人跨境支付场景。

主要功能：

- 注册账户
- 登录系统
- 修改个人资料
- 完成 KYC 认证
- 绑定银行卡
- 申请虚拟卡或实体卡
- 查看多币种钱包
- 发起跨境转账
- 进行货币兑换
- 查看交易记录
- 查看手续费与汇率报价
- 管理收款人
- 申请退款或提交工单

### 3.2 企业用户

企业用户适用于 B2B 支付、批量付款、供应商付款等场景。

主要功能：

- 企业账户注册
- 企业 KYC/KYB 审核
- 添加企业成员
- 设置成员权限
- 批量付款
- API 接入
- 对账文件下载
- 发票与账单管理

### 3.3 管理员

管理员用于系统运营、风控、审核和运维管理。

主要功能：

- 查看用户列表
- 审核用户 KYC
- 审核银行卡申请
- 审核大额交易
- 冻结/解冻用户账户
- 管理交易状态
- 配置汇率加点
- 配置手续费规则
- 配置风控规则
- 查看审计日志
- 查看清算和对账状态
- 处理异常交易
- 生成运营报表

### 3.4 超级管理员

超级管理员拥有系统最高权限，但应该受到更严格的审计和双人审批机制约束。

主要功能：

- 创建管理员账号
- 分配管理员角色
- 配置系统级参数
- 配置支付通道
- 配置清算账户
- 管理密钥和安全策略
- 查看全局审计日志

### 3.5 系统服务账号

系统服务账号不面向人工登录，而用于服务间调用。

主要用途：

- 交易引擎调用账本服务
- 清算服务调用支付通道
- 风控服务接收交易事件
- 通知服务发送交易状态通知
- 对账服务生成对账任务

## 4. 权限模型设计

推荐使用 RBAC + ABAC 混合模型。

### 4.1 RBAC：基于角色的权限控制

```text
User
 |
 v
UserRole
 |
 v
Role
 |
 v
RolePermission
 |
 v
Permission
```

示例角色：

| 角色 | 说明 |
|---|---|
| user_personal | 个人用户 |
| user_business_owner | 企业主账号 |
| user_business_member | 企业成员 |
| admin_kyc | KYC 审核管理员 |
| admin_risk | 风控管理员 |
| admin_ops | 运营管理员 |
| admin_finance | 财务/清算管理员 |
| super_admin | 超级管理员 |
| service_account | 系统服务账号 |

### 4.2 ABAC：基于属性的权限控制

RBAC 解决“你是谁”，ABAC 解决“在什么条件下可以做什么”。

示例规则：

```text
普通用户只能查看自己的交易。
KYC 未通过用户不能发起跨境支付。
单笔超过 10,000 USD 的交易需要进入人工审核。
管理员不能审核自己创建的交易或账户。
财务管理员只能查看清算和对账数据，不能修改用户 KYC 状态。
```

## 7. 银行卡模块设计

银行卡模块包含两类能力：

1. 用户绑定已有银行卡
2. 用户申请 Aspira Pay 虚拟卡或实体卡

### 7.1 银行卡绑定

用户可以绑定自己的外部银行卡，用于充值、提现或支付。

流程：

```text
用户提交银行卡信息
        |
        v
系统校验卡号格式
        |
        v
调用卡组织/支付通道验证
        |
        v
小额验证或 3DS 验证
        |
        v
绑定成功
```

银行卡信息：

- 持卡人姓名
- 卡号 Token
- 卡组织
- 卡类型
- 发卡国家/地区
- 有效期
- 绑定状态
- 默认支付卡标识

重要安全原则：

- 不直接存储完整卡号
- 使用 Tokenization
- CVV 不落库
- 敏感信息加密存储
- 满足 PCI DSS 思路

### 7.2 银行卡申请

用户可以申请虚拟卡或实体卡。

申请流程：

```text
用户完成 KYC
        |
        v
用户选择虚拟卡/实体卡
        |
        v
系统检查账户风险等级
        |
        v
系统检查发卡地区支持情况
        |
        v
调用发卡服务生成卡
        |
        v
写入 Card Service
        |
        v
用户可在 App 中查看和管理卡
```

### 7.3 卡状态

| 状态 | 说明 |
|---|---|
| pending | 申请中 |
| active | 正常 |
| frozen | 已冻结 |
| lost | 挂失 |
| expired | 已过期 |
| cancelled | 已注销 |

### 7.4 卡功能

- 查看卡片
- 冻结/解冻
- 设置限额
- 设置消费地区
- 设置线上支付开关
- 设置 ATM 提现开关
- 查看卡交易记录
- 申请补卡
- 注销卡片

### 7.5 卡支付交易流程

```text
用户使用 Aspira Card 消费
        |
        v
卡组织/支付通道发送授权请求
        |
        v
Card Service 接收授权
        |
        v
Risk Service 做风控判断
        |
        v
Wallet Service 检查余额
        |
        v
FX Service 判断是否需要换汇
        |
        v
Transaction Engine 冻结资金
        |
        v
返回授权成功/失败
        |
        v
清算日完成最终扣款
```

- frozen_balance：冻结余额
- total_balance：总余额
- currency：币种
- status：账户状态

余额变化必须通过账本服务完成，不能直接修改余额字段。

## 9. 账本系统设计

金融系统中，账本比普通余额表更重要。推荐使用双分录账本。

### 9.1 双分录账本

每一笔资金变化都至少有两条记录：

```text
借方账户 Debit
贷方账户 Credit
```

示例：用户 USD 钱包支付 100 USD 手续费 1 USD：

```text
Debit: 用户 USD 钱包              101 USD
Credit: 收款方待清算账户          100 USD
Credit: 平台手续费收入账户          1 USD
```

### 9.2 账本要求

- 不允许物理删除账本记录
- 所有变更追加写入
- 每笔交易有唯一 transaction_id
- 每条账本有唯一 ledger_entry_id
- 支持幂等
- 支持审计回放
- 支持日终对账
- 支持异常冲正

## 11. 汇率与手续费设计

### 11.1 汇率服务

FX Service 负责：

- 获取外部汇率
- 缓存汇率
- 生成用户报价
- 设置报价有效期
- 记录报价快照
- 处理汇率波动风险

报价信息：

- source_currency
- target_currency
- mid_market_rate
- platform_markup
- final_rate
- quote_expire_time

### 11.2 Wise 风格手续费模型

手续费可以拆成：

```text
总手续费 = 固定手续费 + 比例手续费 + 通道成本 + 风险加价
```

示例：

```text
transfer_fee = fixed_fee + amount * variable_rate + channel_cost + risk_markup
```

系统需要支持按以下维度配置：

- 币种
- 国家/地区
- 支付方式
- 收款方式
- 用户等级
- 交易金额区间
- 风险等级
- 促销优惠

## 12. 清算与结算设计

### 12.1 清算模块

Clearing Service 负责把用户交易转化成外部支付网络可识别的清算指令。

支持通道：

- 本地银行转账
- SWIFT
- Visa/Mastercard
- 第三方支付通道
- 合作银行 API
- 内部资金池

### 12.2 结算模块

Settlement Service 负责：

- 清算批次生成
- 通道返回文件解析
- 交易状态更新
- 内部账户对账
- 银行账户对账
- 异常交易处理
- 失败交易退款
- 手续费入账

## 13. 风控系统设计

Risk Service 需要覆盖账户风险、交易风险、设备风险和通道风险。

### 13.1 风控维度

- 用户 KYC 状态
- 用户风险等级
- 登录设备
- IP 地址
- 国家/地区
- 单笔金额
- 日累计金额
- 月累计金额
- 收款人风险
- 交易频率
- 历史拒付记录
- 黑名单/灰名单
- 高风险币种或通道

### 13.2 风控动作

| 动作 | 说明 |
|---|---|
| allow | 放行 |
| reject | 拒绝 |
| review | 人工审核 |
| limit | 限额 |
| freeze | 冻结账户 |
| step_up_auth | 要求二次验证 |

## 16. 微服务划分建议

| 服务 | 推荐语言 | 说明 |
|---|---|---|
| api-gateway | Go | API 网关、路由、认证、限流 |
| user-service | Go | 用户注册、资料、登录 |
| auth-service | Go | JWT/OAuth2/MFA |
| kyc-service | Go | KYC/AML 流程 |
| card-service | Go | 银行卡绑定、申请、状态管理 |
| wallet-service | Go | 钱包账户、余额查询 |
| fx-service | Go | 汇率、报价 |
| payment-service | Go | 支付订单、手续费 |
| transaction-engine | C++/Go | 核心交易状态机 |
| ledger-service | C++/Go | 双分录账本 |
| clearing-service | Go | 清算指令 |
| settlement-service | Go | 结算、对账 |

推荐思路：

- 高并发、强一致交易模块可优先使用 C++ 或 Go。
- API、管理后台、业务服务优先使用 Go。
- 风控模型如未来加入机器学习，可单独使用 Python 服务。

## 18. 高可用与容灾设计

### 18.1 高可用

- API Gateway 多实例
- 核心服务多副本
- PostgreSQL 主从或分布式集群
- Redis Sentinel/Cluster
- Kafka 多 Broker
- 服务熔断和限流
- 支付通道多路由
- 清算任务可重试

### 18.2 容灾

- 数据库定时备份
- 异地备份
- 账本追加写入
- 交易事件进入 Kafka
- 失败任务可重放
- 关键服务多可用区部署

## 19. 安全设计

### 19.1 通信安全

- 全站 HTTPS/TLS
- 服务间 mTLS
- API 签名
- 请求防重放
- IP 白名单
- WAF 防护

### 19.2 数据安全

- 密码使用 Argon2/bcrypt 哈希
- 敏感信息 AES 加密
- 卡号 Token 化
- CVV 不落库
- KYC 文件加密存储
- 数据库字段级加密
- 密钥托管到 KMS/HSM

### 19.3 操作安全

- 管理员 MFA
- 高危操作二次确认
- 高危操作双人审批
- 所有操作审计
- 权限最小化
- 定期权限复核
